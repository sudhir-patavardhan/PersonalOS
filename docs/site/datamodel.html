PersonalOS Data Model

Full schema reference — entities, relationships, storage locations, and privacy notes. PersonalOS never reads raw transaction data; it holds only encrypted pointers.

PersonalOS DB (PostgreSQL)
Arweave (encrypted, permanent)
Base chain (on-chain)
iOS device / iCloud Keychain (ephemeral)

Table-by-Table Schema

souls PersonalOS DB
  • id uuid PK
  • email text UNIQUE
  • arweave_ledger_hash text
  • arweave_account_model enum (shared | soul_owned)
  • arweave_wallet_address text nullable
  • wallet_address text
  • depth_score numeric(6,3)
  • created_at timestamptz
arweave_ledger_hash is a pointer to the encrypted Ledger on Arweave — PersonalOS never stores or decrypts the underlying Transaktions.

arweave_account_model determines whose Arweave wallet holds the Ledger. shared (default): PersonalOS writes via a single Irys account — batched, low cost (~$100/yr at 100k Souls). soul_owned (opt-in): PersonalOS writes to the Soul's own Arweave wallet address, stored in arweave_wallet_address — higher per-transaction cost but full transferability and platform-independent ownership.
wallet_address is the Soul's Coinbase Smart Wallet on Base. Balance state lives on-chain only.
konnections PersonalOS DB
  • id uuid PK
  • soul_id uuid FK → souls
  • provider text (plaid | gmail)
  • status enum
  • plaid_access_token text
  • last_harvest_at timestamptz
plaid_access_token is encrypted at rest (envelope encryption via KMS). Never logged or transmitted in plaintext.
Status values: established · syncing · disconnected
transaktions Arweave only
  • id uuid (within payload)
  • konnection_id uuid (within payload)
  • type text (bank_transaction | email)
  • source text
  • ts timestamptz
  • arweave_tx_id text (Arweave TxID)
Immutable. Stored in encrypted batches on Arweave. PersonalOS holds only the Arweave content hash on the Soul record. This entity does not exist in the PostgreSQL database.
signal_types PersonalOS DB
  • id uuid PK
  • name text UNIQUE (e.g. automotive.new_vehicle_purchase)
  • version text (semver)
Owned and versioned by PersonalOS. Brands target a Signal Type when creating a Listing; Souls grant Consents per Signal Type.
insights PersonalOS DB
  • id uuid PK
  • soul_id uuid FK → souls
  • signal_type_id uuid FK → signal_types
  • score numeric(5,4)
  • algorithm_version text (semver)
  • is_stale boolean
  • computed_at timestamptz
Derived propensity score. Computed on-device during Scoring — only scores leave the device, never raw Transaktions. algorithm_version tracks which Scoring algorithm produced this score. is_stale is set true when a new algorithm ships and re-scoring is pending; the Exchange continues serving stale scores until re-scoring completes in the background.
convictions PersonalOS DB
  • id uuid PK
  • soul_id uuid FK → souls
  • signal_type_id uuid FK → signal_types
  • score numeric(5,4)
  • last_purchase_at timestamptz NULLABLE
  • decay_reset_at timestamptz
  • updated_at timestamptz
Per-Signal-Type real-buyer track record. Computed on-device during Scoring — only the score reaches the server. Used in Exchange ranking: bid × Conviction × recency_weight.
Score decays by 50% every 6 months of inactivity (no matching purchase detected in the Signal Type's conversion window). Never decays to zero. decay_reset_at is updated when a matching purchase Transaktion is detected in a Harvest. Conversion windows are per-Signal-Type and published in the open Scoring algorithm.
consents PersonalOS DB
  • id uuid PK
  • soul_id uuid FK → souls
  • signal_type_id uuid FK → signal_types
  • yield_floor_usdc numeric(12,6)
  • notification_max_per_day int (default 1)
  • notification_quiet_hours text (e.g. "22:00-07:00")
  • created_at timestamptz
  • revoked_at timestamptz NULLABLE
Active consent: revoked_at IS NULL. yield_floor_usdc is a hard pre-ranking gate — the Exchange never delivers an Offer below this value. notification_max_per_day and notification_quiet_hours are Soul-configured caps the Exchange respects as hard limits before push delivery. Default on signup: 1 push/day, no quiet hours.
brands PersonalOS DB
  • id uuid PK
  • name text
  • subscription_status enum
  • stripe_subscription_id text
stripe_subscription_id is treated as a sensitive billing reference — access is restricted to billing service only.
listings PersonalOS DB
  • id uuid PK
  • brand_id uuid FK → brands
  • signal_type_id uuid FK → signal_types
  • bid_per_claim_usdc numeric(12,6)
  • content jsonb (creative)
  • voucher_delivery_webhook text NULLABLE
  • status enum
  • on_chain_escrow_tx_hashtext
  • budget_remaining_usdc numeric(16,6)
Status values: pending_review · active · exhausted
on_chain_escrow_tx_hash is a pointer to the escrow event on Base. Budget is funded directly in USDC via Coinbase Commerce — PersonalOS never handles fiat on the Brand side. Budget state of record is on-chain; budget_remaining_usdc is a denormalized cache updated by event listeners.
voucher_delivery_webhook (optional): if set, PersonalOS calls this URL on each Claim and the Brand returns a fresh single-use code. If null, Vouchers in content are non-unique per Soul — Voucher integrity is the Brand's responsibility.
offers PersonalOS DB
  • id uuid PK
  • soul_id uuid FK → souls
  • listing_id uuid FK → listings
  • status enum
  • ranking_score numeric(10,6)
  • push_sent_at timestamptz NULLABLE
  • first_seen_at timestamptz NULLABLE
  • expiry_at timestamptz
  • delivered_at timestamptz NULLABLE
Status lifecycle: pending → delivered → seen → claimed | dismissed | expired. Created by the Exchange (event-driven, no persistent Exchange entity). ranking_score = bid × Conviction × recency_weight at match time.
expiry_at is computed dynamically: 72h from push_sent_at if a push was sent; otherwise 72h from first_seen_at. Silent queued Offers (behind notification cap) do not start expiring until they surface to the Soul.
claims PersonalOS DB
  • id uuid PK
  • offer_id uuid FK → offers
  • usdc_gross numeric(12,6)
  • platform_fee_usdc numeric(12,6)
  • yield_usdc numeric(12,6)
  • chain_tx_hash text
  • claimed_at timestamptz
chain_tx_hash references the settlement event on Base. yield_usdc = usdc_gross − platform_fee_usdc. On-chain state is authoritative; DB row is an audit mirror.
withdrawals PersonalOS DB
  • id uuid PK
  • soul_id uuid FK → souls
  • amount_usdc numeric(16,6)
  • coinbase_offramp_ref text
  • status enum
  • initiated_at timestamptz
Soul moves USDC from their non-custodial Base Wallet to fiat via Coinbase offramp. coinbase_offramp_ref is the external transaction reference.
wallet Base chain
  • wallet_address address (EVM)
  • usdc_balance uint256 (on-chain)
  • chain Base mainnet
Non-custodial Coinbase Smart Wallet. PersonalOS stores only wallet_address on the souls row. USDC balance and transfer history live on-chain only — PersonalOS reads via event indexer.
