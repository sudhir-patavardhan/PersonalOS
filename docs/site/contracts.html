Trust Architecture

From promise
to proof.

Every other platform asks you to trust its intentions. PersonalOS replaces that trust requirement with code deployed on Base that neither PersonalOS nor anyone else can override after launch.

The fee split is not a policy PersonalOS can change tomorrow — it is a constant frozen at deploy time. The Budget escrow is not a promise to hold funds safely — it is a constraint the contract enforces structurally. The Yield deposit is not a transfer request PersonalOS initiates — it is an atomic settlement the contract executes directly to the Soul's Wallet the moment a Claim is made.

PersonalOS cannot redirect it, withhold it, or change the split. The rules are in the code, the code is on a public chain, and anyone can audit it. This is what data sovereignty looks like at the infrastructure level.

⛓
Immutable by design
Core contracts are deployed once and never upgraded. The fee rate, settlement logic, and escrow rules are frozen at deploy time — PersonalOS cannot change them unilaterally. Brands and Souls can verify this on-chain.
⚡
Atomic settlement
A Claim either settles completely — fee to PersonalOS, Yield to the Soul's wallet — or it doesn't happen at all. There is no partial settlement, no processing lag, no credit risk. The smart contract is the payment processor.
🔍
Publicly auditable
Every Budget deposit, every Claim settlement, every Yield payment is a transaction on Base chain. Anyone can read the contract ABI, verify the fee split, and trace every USDC movement. No black box.
🔐
Non-custodial by default
PersonalOS cannot access a Soul's Wallet. The Soul's passkey controls the Smart Wallet — PersonalOS can only deposit Yield into it via the settlement contract. Funds flow in, but PersonalOS has no outbound access.

The Contracts

BudgetEscrow.sol
Holds Brand USDC from Listing creation through Claim settlement
Base mainnet Immutable USDC ERC-20
Brand's Budget is locked from the moment of deposit. PersonalOS cannot withdraw it. It only moves in two ways: atomically to a Soul + PersonalOS on a valid Claim, or back to the Brand when a Listing expires with remaining Budget.
Key functions
  • deposit(listing_id, amount_usdc)
  • settle(claim_id, soul_address, gross_usdc)
  • refund(listing_id)
  • feeRate() → uint256 // immutable
SoulWallet
Coinbase Smart Wallet — one per Soul, passkey-controlled
Base mainnet Per Soul ERC-4337
PersonalOS can only deposit into this wallet via BudgetEscrow.settle(). It cannot initiate withdrawals, freeze funds, or redirect Yield. The Soul's passkey — backed by iCloud Keychain — is the only key that controls outbound transfers.
Key functions
  • receive() // accepts USDC from settlement
  • execute(dest, value, data) // Soul-initiated only
  • validateUserOp(op, hash, missingFunds)
  • exportOwnership() // Soul portability
ConsentRegistry.sol
On-chain record of Soul Consents per Category — roadmap
Base mainnet Roadmap
Today, Consents live in the PersonalOS database. The roadmap moves them on-chain: a Soul's opt-in, Yield floor, and revocation are recorded on Base, making PersonalOS's matching engine provably Consent-gated — auditable by Brands, regulators, and Souls alike.
Planned functions
  • grantConsent(category, yield_floor_usdc)
  • revokeConsent(category)
  • getConsent(soul, category) → ConsentRecord
  • hasActiveConsent(soul, category) → bool

What the contracts enforce

PersonalOS cannot
  • Change the fee rate after a contract is deployed — it is a compile-time constant
  • Access Soul Wallet funds — the settlement contract can only deposit, never withdraw
  • Redirect Yield away from the Soul's wallet address on a valid Claim
  • Double-charge a Claim — claim_id idempotency key prevents re-execution
  • Freeze a Brand's Budget without a valid Claim or Listing expiry
  • Settle without a matching Claim — the Exchange backend must call settle() with a valid claim_id
PersonalOS can
  • Call settle() on each valid Claim — this is the only write PersonalOS makes to the escrow contract
  • Review Listing content before activation — off-chain moderation, on-chain Budget is unaffected
  • Deploy new contract versions with updated fee rates — new Listings use the new contract, existing Listings honour the original
  • Publish the contract address and ABI — this is a commitment, not a courtesy
  • Call refund() when a Listing expires with remaining Budget — returns funds to the Brand

Verify it yourself

Contract address
Published at launch on Base mainnet. Any Base block explorer can verify the bytecode, ABI, and all historical transactions.
Fee rate verification
Call BudgetEscrow.feeRate() at any time. The returned value is the immutable fee split. It cannot differ from what was set at deploy time.
Claim audit trail
Every settle() call emits a ClaimSettled event on-chain. Filter by Soul wallet address to see every Yield payment ever made, independently of PersonalOS.
